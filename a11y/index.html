More than screenreaders

  • Blindness
  • Colour blindness
  • Cognitive impairements
  • Motion and hearing impairements
  • Vestibular disorders
  • Many more …
There are many more shades to accessibility other than just making your application accessible to screenreaders. Though please do make them accessible to screenreaders :)

"Slow Keys"

A keystroke is only registered after it was held for a certain period of time. Helps people with motor impairments with unpredictable motions.

Please don't disable the zoom 


<meta name="viewport" content="
	width=device-width,
	initial-scale=1,
	maximum-scale=1,
	user-scalable=no
">


@-ms-viewport { width: device-width; zoom:1.0; }
That's just mean! I got glasses and I'd appreciate it if I could zoom in please. WCAG 1.4.4

Parallax can cause vertigo / motion sickness

It's also ennoying to get your scroll hijacked.

Outlines

Don't remove. Just hide with JS and add when tab is first pressed.

Color AND shape

Make distinctions with both. E.g. Links.

SVG accessibility 


<svg aria-labelledby="title-download">
	<title id="title-download" lang="en">Download symbol</title>
	<path class="icons-background" d="M27.8 etc…"></path>
</svg>


<svg role="img" title="Download symbol">
	<use xlink:href="sprite.svg#download"/>
</svg>
In a sprite use the title attribute and tie it to the SVG via aria-describedby

Slow internet

  • Remote areas in Australia
    (Almost everywhere in Oz really)
  • Remote areas not in Australia
    (Almost everywhere else really)
  • Train / Bus in tunnel
  • Old contract only includes EDGE connection
  • Student in a library
  • Public wifi
Slow internet is common and very much an accessibility issue.

User feedback

Good feedback is important so users know what's going on. This nice article at smashing magazine describes how to make your messages accessible. You have a maximum of 10 sec of your users attention according to this Nielsen article, fill it with usefull feedback and give up at some point. Add fallbacks to your nodejs app with a nginx proxy and an error page.

Offline first

    • W3C Specifications
      (awesome GitHub repo)

      • Offline Web Applications caching, events, localStorage
      • Service Workers
      • IndexedDB
      • WebStorage

    • A LIST APART article

      • Remain functional and store locally, add to a queue
      • Offline is not an error.
      • Commuicate a possible risk of something being out of sync
Build your app offline first. Have a look at this repo about all things offline first. There is a fantastic repo for more docs under: https://github.com/pazguille/offline-first.

Separation of concerns

  • Keep style and content separated
  • HTML (no CSS) is the best fallback
  • Progressive enhancements
When building your SPA or website make sure you devide and conquer. When HTML is loaded it should be functional and readable. HTML is your best fallback.

HTTP/2

  • No more http request counting
  • Modularize all the things
  • Inlining critical CSS = Anit Pattern
HTTP/2 and SPDY article is a great resource.
